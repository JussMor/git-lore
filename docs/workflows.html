Core Flow: Init to Accepted Lore

1) init

git-lore init .

Creates .lore layout for active intent, checkpoints, signals, and accepted refs.

2) mark / propose

git-lore mark --title ...

Records new lore atoms in proposed state as typed rationale units.

3) checkpoint

git-lore checkpoint .

Snapshots the current atom set and prepares commit-trailer-ready narrative.

4) signal

git-lore signal --path ...

Broadcasts intent to reduce collisions before branch merge pain appears.

5) commit

git-lore commit --message ...

Runs validation, commits lore trailers, then accepts active non-deprecated atoms.

6) sync / explain

git-lore sync | explain

Rehydrates context from history and exposes rationale lineage for active edits.

State Machine

DRAFT -> PROPOSED -> ACCEPTED -> DEPRECATED (local) (active) (committed) (historical)

Live metrics in this page are computed from .lore files and grouped by these states.

Conflict Semantics

TypeConflict

Same logical location, incompatible rationale updates. High entropy.

DependencyConflict

One branch deprecates lore while another still depends on it.

Additive Merge

Independent rationale additions can merge safely.

PRISM Signaling Loop

1. Agent emits session signal 2. Signal written under .lore/prism 3. Existing signals scanned for overlap 4. Soft-lock warning or hard-lock violation

Soft-lock overlap is path-centric today. Hard-lock checks evaluate conflicting decisions against active lore atoms.
